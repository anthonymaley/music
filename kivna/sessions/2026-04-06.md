## What Was Done

- **AirPlay ghost connection fix**: Replaced per-speaker wake cycle (deselect one → 500ms → reselect) with full AirPlay reset (snapshot all active non-local speakers + volumes → deselect all → 1.5s → reselect all → restore volumes → 1.5s). Applied to all play paths including playlist browser, which was previously missing reset entirely.
- **Unified timeline model**: Context-aware TUI now renders full `contextTracks` playlist with history as visual overlay (played tracks dimmed). Standalone TUI uses history-only timeline (removed `pollSurroundingTracks` which was rebuilding to a different album context on every shuffle).
- **TUI navigation overhaul**: Added `<`/`>` and `,`/`.` for track skip (previous/next). Added `b` in playlist browser to jump to Now Playing without starting new playback (symmetric with Now Playing `b` → browser). Removed `l`/`d` love/dislike keybindings.
- **Loved/disliked indicators**: Added ♥ (red) and ↓ (dim) next to track title in Now Playing. Wrapped in inner AppleScript `try` to prevent breaking `pollNowPlaying` on unsupported track types.
- **User fixes**: Simplified play-track-by-index to use `play track N of playlist`, removed speaker wake from play.md slash command.

## Key Decisions

- Full AirPlay reset over surgical per-speaker wake: can't programmatically distinguish "healthy selected" from "ghost selected" — both report `selected: true`. Resetting all is deterministic.
- History as overlay, not structure: context-aware TUI shows the full playlist with played state from `history` array, rather than using history to replace the playlist prefix. This keeps all tracks navigable and is correct under shuffle.
- Standalone TUI uses history-only (no "next"): `pollSurroundingTracks` queries `current playlist` which changes to album/single context under shuffle. History is the only stable data source.
- Volume restore in reset: deselecting speakers can reset volume to system default. Explicit restore of snapshotted volumes prevents level loss.

## Gotchas

- `loved of current track` / `disliked of current track` throws for some track types (radio, streaming). Must be in its own inner `try` block or it kills the entire `pollNowPlaying` — TUI shows "Nothing playing" while music is playing.
- AirPlay reset adds ~3s latency when non-local speakers are active. Acceptable for reliability but noticeable.
- The `,` and `.` keys may not register on all terminal configurations. `<` and `>` (shift variants) are also mapped as fallback.

## Insights

- Code review caught a real architectural issue: inferring "played" from `i < currentIdx` in playlist order is wrong under shuffle (tracks before current index were never actually played). History overlay is the correct model.
- The ghost connection problem was partly a coverage issue — playlist browser play paths bypassed the CLI `Play` command entirely, going straight to AppleScript without any AirPlay reset.
- Two timeline models (context vs standalone) is the right split: context mode has a stable playlist source, standalone mode doesn't. Trying to unify them with `pollSurroundingTracks` was the root cause of the shuffle instability.
